* Scope and Design Invariants

=elfeed-translate= is a modular Emacs Lisp package that translates Elfeed entry
titles and/or content through an OpenAI-compatible chat-completions API.
It generates separate local RSS 2.0 feeds instead of mutating original entries.

The following are intentional invariants:

1. One installation has one global target language.
2. Title and content translation are independently selected by feed tags.
3. A translation is written only after an entire API batch is paired safely.
4. Source feeds and their Elfeed database entries are never overwritten.
5. Transport failures fail fast; only unusable translation results are retried.
6. API credentials must never appear in package-generated diagnostics.

Per-feed languages, runtime language switching, and language-aware cache keys
are not current goals.  A deliberate target-language change requires an
explicit cache clear.

* Source Layout

The package facade and five internal modules live together on =load-path=.
Users continue to load only =(require 'elfeed-translate)=.

| File | Responsibility |
|------+----------------|
| =elfeed-translate.el= | Public commands, setup/teardown and global mode |
| =elfeed-translate-core.el= | Customization and shared structured-result helpers |
| =elfeed-translate-cache.el= | SQLite schema, migration, lookup, cache-key generation and transactional writes |
| =elfeed-translate-api.el= | Request validation, HTTP transport, response parsing and diagnostics |
| =elfeed-translate-elfeed.el= | Elfeed DB/tag access, entry extraction, feed paths, RSS rendering and display helpers |
| =elfeed-translate-engine.el= | Collection, batching, retries, serial/parallel state, fail-fast abort, update hooks and finalization |
| =test/*.el= | Offline ERT tests, one suite per source module, plus the batch runner |

Dependencies are deliberately one-way.  The facade requires the engine and
adapters.  The engine requires core, cache, API and Elfeed adapter; adapters do
not require the engine or facade.  The API does not know SQLite cache keys or
translation-cycle busy state.  RSS rendering receives a translation lookup
function from the engine/facade, so the Elfeed adapter does not require cache.

Dependencies are built into Emacs 29.1 except Elfeed: =url=, =json=, =sqlite=,
=xml=, =subr-x=, =cl-lib= and =seq=.

* End-to-End Lifecycle

#+begin_src text
global-elfeed-translate-mode
  -> elfeed-search-mode-hook
     -> elfeed-translate-setup
        -> open/create SQLite cache
        -> generate initial local RSS files
        -> install elfeed update hooks

elfeed-update
  -> elfeed-update-init-hooks records expected feed count
  -> elfeed-update-hooks counts completed feeds
  -> elfeed-translate--on-all-feeds-updated
     -> elfeed-translate--on-db-update
        -> collect uncached title and content items
        -> deduplicate source text
        -> split title and content batches independently
        -> merge them into one queue
        -> serial or bounded-parallel dispatch
        -> transactional cache write for every successful batch
        -> regenerate affected local RSS files
        -> optional guarded auto-refresh
#+end_src

The title queue uses =elfeed-translate-system-prompt= and
=elfeed-translate-batch-size=.  The content queue uses the content prompt and
content batch size.  Queue elements have this shape:

#+begin_src elisp
(:call-fn elfeed-translate--call-api
 :texts ("source 1" "source 2")
 :prompt PROMPT-TEMPLATE
 :retries 0)
#+end_src

* Request Construction and Encoding

=elfeed-translate--build-request= is the only production request builder.
=elfeed-translate-test-api= calls the same builder and transport path.

The builder:

1. Assigns =item-0001=, =item-0002=, ... within each batch.
2. Serializes the user message as an id-bearing JSON array.
3. Embeds that text in the OpenAI chat-completions request object.
4. Serializes and parses the outer JSON once as a validation step.
5. Requires the final body to be a UTF-8 unibyte string.
6. Trims, validates and converts the API key and all headers to ASCII unibyte.

The header normalization is required by Emacs =url-http=.  A multibyte ASCII
Authorization value combined with raw UTF-8 JSON can promote the complete HTTP
request to a multibyte string; =url-http-create-request= rejects that request
before it is sent.

The request =Content-Type= is =application/json; charset=utf-8=.  Debug output
may report byte counts and multibyte state but never the Authorization value.

* Id-Bearing Batch Protocol

Input is still ordinary text delivered in =messages[1].content=.  Its syntax is:

#+begin_src json
[
  {"id":"item-0001","text":"source text"},
  {"id":"item-0002","text":"another source"}
]
#+end_src

Expected model output:

#+begin_src json
[
  {"id":"item-0002","translation":"translated second item"},
  {"id":"item-0001","translation":"translated first item"}
]
#+end_src

Pairing is by id, not array position.  The parser requires every expected id
exactly once and rejects unknown, missing and duplicate ids.  This makes model
reordering harmless and allows source/translation text to contain =---=,
quotes, newlines, backslashes or HTML without delimiter ambiguity.

The parser accepts a top-level array and, defensively, an object containing a
=translations= array.  It also removes a surrounding Markdown code fence.  If
content is not JSON at all, the old =---= and line parsers run as compatibility
fallbacks.  Valid JSON with invalid ids does not fall back: it is a protocol
failure and is retried.

* Structured API Results

Callbacks receive a plist rather than a pairs-or-nil value.

Success contains source-text/translation pairs; MD5 keys are computed only by
the cache module:

#+begin_src elisp
(:ok t
 :pairs (("source text" . "translation"))
 :http-status 200
 :finish-reason "stop"
 :protocol id-json)
#+end_src

Failure:

#+begin_src elisp
(:ok nil
 :kind output-mismatch
 :message "Expected 10 translations, received 9"
 :retryable t
 :http-status 200
 :finish-reason "stop")
#+end_src

The HTTP callback status plist is inspected before response parsing.  Timeout
logic stores the buffer returned by =url-retrieve= immediately, so it can stop
the process even before a response callback fires.  A shared =done= flag
prevents late responses from invoking the batch callback twice.

* finish_reason and Retry Classification

=finish_reason= is recorded on both successful and failed results and displayed
by the API test command.  Classification is intentionally biased toward fast
failure for external transport state:

| Condition | Retry? | Failure kind |
|-----------+--------+--------------|
| URL transport error, proxy/TLS failure, timeout, empty response | No | =network=, =timeout=, =empty-response= |
| Any HTTP error, including 429 and 5xx | No | =http= |
| Missing =choices[0].message.content= | No | =api-structure= |
| HTTP 2xx response JSON cannot be parsed | Yes | =json= |
| =finish_reason= =stop= or absent | Continue parsing | — |
| =finish_reason= =length= or =max_tokens= | Yes | =completion-truncated= |
| =finish_reason= =content_filter= or =safety= | No | =completion-filtered= |
| Other non-stop finish reason | Yes | =completion-incomplete= |
| Invalid id JSON, missing/duplicate/unknown id | Yes | =translation-json= |
| Legacy output count mismatch | Yes | =output-mismatch= |

This policy is not a general-purpose HTTP retry strategy.  It reflects the
interactive feed workflow: return control quickly when the network/provider is
unavailable, but give the model a limited chance to correct an otherwise valid
translation request.

Retryable failures use exponential backoff with up to 25 percent jitter.  The
delay is capped by =elfeed-translate-retry-max-delay=.  Serial and parallel
dispatchers both hold =elfeed-translate--busy= across delayed retries.

Fatal cycle-wide failures (=network=, =timeout=, =send=, =http=,
=empty-response= and request/configuration failures) clear every unsent batch
and cancel delayed retries.  Requests already in flight are drained.  The
engine then writes any successful results, regenerates RSS once, clears busy
state and reports completed/failed/skipped counts.

Parallel state lives in =elfeed-translate--parallel-state=, including
=:queue=, =:in-flight=, =:retry-waiting=, =:completed= and =:total=.  Delayed
retries increment =:retry-waiting= so an empty queue cannot finalize early.
Top-level callback/timer functions are used instead of self-referential local
closures for reliable =eval-buffer= behavior.

* SQLite Translation Cache

The cache file is =translate-cache.sqlite= under
=elfeed-translate-output-dir= (by default
=<elfeed-db-directory>/translated=).  Schema:

#+begin_src sql
CREATE TABLE translations (
  key   TEXT PRIMARY KEY,
  value TEXT NOT NULL
);
#+end_src

=key= is =MD5(source-text)= and =value= is the raw translation.  This is
consistent with the single-target-language invariant.  A target-language
change without clearing the cache is unsupported.

Each successful batch is written in one SQLite transaction.  Cache persistence
is therefore incremental and crash-safe.  The legacy
=translate-cache.el= hash table is migrated once and renamed with a
=.migrated= suffix.

Title display style is not part of the cache.  Formatting is applied when RSS
is generated, so changing title order or separator only requires regeneration.

* Local RSS Generation

Each source URL maps to =MD5(feed-url).xml= under
=elfeed-translate-output-dir=.  Item GUIDs use
=MD5(feed-url):original-entry-id= to prevent collisions between local feeds.

Title and content translations are read independently.  When a translation is
not yet cached, generated entries retain the available original title/content
instead of disappearing from the local feed.

* Diagnostics

=elfeed-translate-test-api= sends two titles through the full production path.
Its report contains:

- endpoint, model and target language;
- JSON byte/multibyte and header checks;
- HTTP status and elapsed time;
- =finish_reason= and selected parser protocol;
- source and translated text, or structured failure metadata;
- a permanently redacted API key field.

Failed raw responses are appended to =*elfeed-translate-debug*= with timestamp,
reason and source-item count.  HTTP provider error JSON is summarized when
possible.

* Critical Implementation Notes

1. Always pass =:object-type=, =:array-type=, =:null-object= and
   =:false-object= directly to =json-parse-string=.
2. The API module must not read or mutate =elfeed-translate--busy=.  Serial and
   parallel dispatch in the engine own the lock for the complete cycle.
3. Do not replace direct =url-retrieve= with =url-queue-retrieve=: delayed
   execution loses dynamic request bindings.
4. Preserve the timeout =done= guard to prevent double callbacks.
5. The system prompt templates contain exactly one =%s= placeholder for the
   global target language.
6. Preserve id validation before cache writes.  Partial batches must never be
   cached by position.
7. Do not print or dump Authorization headers.
8. Keep RSS output explicitly bound to =utf-8-unix=; its XML declaration says
   UTF-8 and Windows otherwise may use the locale code page.
9. Never leave a dangling =;;;###autoload= cookie before a module's =provide=
   form.  Straight's autoload generator will copy that =provide= into the
   autoload file, causing later =require= calls to skip the real module while
   its function definitions remain unloaded.

* Validation

Run the committed offline suite with Elfeed on =load-path=:

#+begin_src shell
emacs --batch -Q -L . -L test -L /path/to/straight/build/elfeed \
  -l test/run-tests.el
#+end_src

The suite covers every source module: request encoding and parsing, id pairing,
=finish_reason=, isolated SQLite persistence, injected RSS lookup and UTF-8
output, module ownership, collection and batching.  Before releasing changes:

1. Run the ERT suite and =check-parens= for every source file.
2. Byte-compile all source and test modules with Elfeed on =load-path=.
3. Optionally start a temporary daemon and use =emacsclient= to require the
   facade and confirm =symbol-file= ownership for each internal function.
4. Exercise serial and parallel delayed retries and confirm the busy lock and
   parallel state are cleared exactly once.
5. Run =M-x elfeed-translate-test-api= against the configured provider.
6. Run an Elfeed update twice and verify the second cycle is a cache hit.
