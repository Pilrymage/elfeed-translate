* Architecture Overview

  =elfeed-translate= is a single-file Emacs Lisp package (~920 lines)
  that translates Elfeed RSS entry titles using any OpenAI-compatible
  LLM API.  It generates local RSS 2.0 XML files as /separate
  subscription sources/, leaving the original feeds untouched.

  The package is organized into these logical sections (in file
  order):

  | Section              | Lines   | Purpose                                                         |
  |----------------------+---------+-----------------------------------------------------------------|
  | Customization        | ~42–174 | =defgroup= + 14 =defcustom= options                              |
  | Translation Cache    | ~175–229| Hash-table cache with disk persistence                          |
  | Utility Functions    | ~230–298| Feed hashing, file paths, GUID generation, autotag helpers       |
  | RSS XML Generation   | ~299–370| RSS 2.0 XML output, title formatting                            |
  | API Client           | ~371–562| HTTP request/response, JSON parsing, batch translation           |
  | Core Translation     | ~563–578| Collecting untranslated titles across tagged feeds               |
  | Feed List Display    | ~579–665| =show-feeds= buffer (org / Elisp format)                         |
  | DB Update Handler    | ~666–751| Hook integration, batch splitting, async processing chain        |
  | Public Commands      | ~752–888| =setup=, =teardown=, =update=, =clear-cache=, =stats=            |
  | Global Minor Mode    | ~889–915| =global-elfeed-translate-mode=, =elfeed-search-mode-hook=         |

  M-x elfeed
    └─ elfeed-search-mode-hook
         └─ elfeed-translate-setup()       ★ 幂等；首次加载缓存 / 生成初始 RSS

  M-x elfeed-update
    └─ Elfeed fetches all feeds in elfeed-feeds
         └─ Entries parsed & added to elfeed-db
              └─ elfeed-db-update-hook fires
                   └─ elfeed-translate--on-db-update()
                        │
                        ├─ 检查 busy 锁 → 如果忙则跳过
                        ├─ (elfeed-translate--translatable-feeds)  → 获取标记了 translate-title 的 feed URL
                        ├─ (elfeed-translate--collect-untranslated) → 遍历每个 feed 的条目，检查缓存
                        │     缓存 key = (elfeed-entry-title entry)  即原始标题
                        │     缓存命中 → 跳过
                        │     缓存未命中 → 加入待翻译列表
                        │
                        ├─ 去重 → 分批 (elfeed-translate-batch-size, 默认 30)
                        ├─ (elfeed-translate--process-batches)        ★ 递归异步链

                        │      │    └─ url-retrieve → 异步等待响应
                        │      │         └─ 回调: 解析 JSON → 存入缓存 → 递归调用 process-batches(batch2, ...)
                        │      │
                        │      ├─ Batch 2: call-api(titles2, callback)
                        │      │    └─ ...同上...
                        │      │
                        │      └─ 所有批次完成:
                        │           ├─ (elfeed-translate--save-cache)    ← 持久化到磁盘

                        │

    └─ Elfeed 抓取所有 feed（包括翻译后的本地 file:// 订阅）
         └─ 翻译标题出现在搜索列表中（带 translated 标签）

* Key Data Structures

  ** Translation Cache

     #+begin_src elisp
       (defvar elfeed-translate--cache (make-hash-table :test 'equal))
       ;; Key:   original title string (from elfeed-entry-title)
       ;; Value: translated title string
     #+end_src

     Persisted to =~/.elfeed/translated/translate-cache.el= as a
     printed hash-table (readable by =read=).  Uses =equal= test so
     string keys work correctly.

     The cache is saved /only/ after all API batches complete.  If
     Emacs crashes mid-translation, unsaved batches are lost (but this
     is acceptable — they'll be re-translated on next update).

  ** Feed-to-File Mapping

     Each original feed URL → one local RSS file:

     #+begin_src elisp
       (defun elfeed-translate--local-feed-path (feed-url)
         (let ((hash (secure-hash 'md5 feed-url))
               (dir  (expand-file-name elfeed-translate-output-dir)))
           (expand-file-name (concat hash ".xml") dir)))
     #+end_src

     The MD5 hash is /stable/ — the same URL always produces the same
     filename, so users can safely hard-code file:// URLs in their
     config.

  ** Title-to-Feed Reverse Index

     During batch processing, a temporary hash table maps each
     /original title/ to the list of feed URLs that contain that
     title:

     #+begin_src elisp
       (make-hash-table :test 'equal)
       ;; Key:   original title string
       ;; Value: list of feed-url strings
     #+end_src

     This handles the case where the same title appears in multiple
     feeds (e.g., cross-posted articles).  After translation, all
     affected feeds' RSS files are regenerated.

* API Client Details

  ** Request Format (OpenAI-compatible)

     #+begin_src js
       {
         "model": "gpt-4o-mini",
         "messages": [
           {"role": "system", "content": "<system-prompt>"},
           {"role": "user",   "content": "title1\n---\ntitle2\n---\ntitle3"}
         ],
         "temperature": 0.3
       }
     #+end_src

     Titles are separated by ="\n---\n"= in the user message.  The
     system prompt instructs the model to return translations
     separated by the same marker.

  ** Response Parsing

     1. Extract HTTP body (skip headers via =re-search-forward
        "^\r?\n"=)
     2. =string-trim= the body
     3. Parse JSON with =json-parse-string= using =:object-type
        'alist= (CRITICAL: must pass as keyword argument, NOT rely on
        dynamic variable binding — passing any keyword like
        =:null-object= causes Emacs to ignore =json-object-type=)
     4. Retry with =:allow-trailing-content t= on parse failure
     5. Navigate =choices[0].message.content=
     6. Split content by ="---"= and pair with input titles in order
     7. Fallback: if counts mismatch, try splitting by newlines

  ** Batch Processing Architecture

     Titles are split into batches of =elfeed-translate-batch-size=
     (default 30).  Batches are processed /sequentially/ via a
     recursive callback chain:

     #+begin_src elisp
       (defun elfeed-translate--process-batches (batches title->feeds total-batches)
         (when batches
           (let ((batch (car batches))
                 (remaining (cdr batches)))
             (elfeed-translate--call-api
              batch
              (lambda (pairs)
                ;; Store results
                (dolist (pair pairs)
                  (elfeed-translate--cache-set (car pair) (cdr pair)))
                ;; Recurse to next batch, or finalize
                (if remaining
                    (elfeed-translate--process-batches remaining ...)
                  (elfeed-translate--save-cache)
                  (elfeed-translate--finalize ...)))))))
     #+end_src

     Each batch's API call is async (=url-retrieve=).  The callback
     stores results and recursively triggers the next batch.  Only
     /after/ all batches complete is the cache saved and RSS
     regenerated.

  ** Concurrency Guard (=elfeed-translate--busy=)

     - Set to =t= before each =url-retrieve=
     - Cleared in the callback /before/ the lambda callback is invoked
       (allowing the next batch in the chain to start)
     - Also cleared in =unwind-protect= cleanup form as safety net
     - =on-db-update= checks =busy= before starting a new translation
       cycle, preventing duplicate API calls if the hook fires during
       an active cycle

  ** Prompt Engineering

     The system prompt (=elfeed-translate-system-prompt=) uses a
     few-shot example:
     - Shows input format (3 titles separated by =---=)
     - Shows expected output format
     - Emphasizes "Do NOT add numbers, bullets, quotes"
     - Instructs: "If already in target language, output unchanged"

     Known sensitivity: models sometimes still add numbering or
     quotes.  The fallback line-by-line parsing helps.  If a non-LLM
     model (e.g., older GLM/Zhipu) is used, consider lowering
     =batch-size= to ~10-15.

* RSS Generation

  ** Output Format

     RSS 2.0 XML with Atom self-link:

     #+begin_src xml
       <?xml version="1.0" encoding="UTF-8"?>
       <rss version="2.0" xmlns:atom="http://www.w3.org/2005/Atom">
         <channel>
           <title>[TL] Original Feed Title</title>
           <link>https://original-feed-url</link>
           <description>Auto-translated RSS feed for ...</description>
           <atom:link href="file:///..." rel="self" type="application/rss+xml"/>
           <item>
             <title>Translated Title</title>
             <link>https://original-entry-link</link>
             <guid isPermaLink="false">feedhash:entryid</guid>
             <pubDate>Tue, 01 Jun 2026 18:00:00 +0000</pubDate>
           </item>
         </channel>
       </rss>
     #+end_src

  ** GUID Scheme

     Entry GUIDs in the generated RSS are composite:
     =<MD5(feed-url)>:<original-entry-id>=.

     This prevents ID collisions when two different original feeds
     happen to have entries with the same GUID (rare but possible).
     Since the translated feed's namespace is =""= (empty, from the
 =file:///= URL), all translated feeds share the same namespace,
     making the composite GUID essential.

  ** Title Formatting

     =elfeed-translate--format-title= applies
 =elfeed-translate-title-style= at RSS generation time /only/.
     The cache always stores the raw translation (no formatting).
     This means:

     - Changing =title-style= or =title-separator= does /not/ require
       cache invalidation
     - Just call =M-x elfeed-translate-update= or regenerate RSS files

* Cache Persistence

  ** File Location

     =~/.elfeed/translated/translate-cache.el=

  ** File Format

     A single Emacs Lisp expression: a hash-table printed by =print=:

     #+begin_src elisp
       #s(hash-table test equal data
          ("Original Title 1" "翻译标题1"
           "Original Title 2" "翻译标题2"))
     #+end_src

  ** Loading

     Read via =read= from a temp buffer.  Validated with
 =hash-table-p=.  On error, the cache is silently reset to empty
     (previous translations are lost but no data corruption occurs).

  ** Dirty Flag

 =elfeed-translate--cache-dirty= is set to =t= by =cache-set=.
     It's cleared by =save-cache=.  =save-cache= is a no-op when
 =dirty= is =nil=.

  ** Known Limitation

     =cache-set= does /not/ store when the translation is identical to
     the original (=equal= returns =t=).  This was intended to avoid
     wasting cache entries on "already in target language" titles, but
     it means those titles are re-scanned on every update.  A future
     fix could cache them with a sentinel value.

* Feed Tag System

  ** How Feeds Are Identified for Translation

     1. =elfeed-translate--translatable-feeds= iterates over
 =elfeed-feeds=
     2. For each entry, extracts the URL (=car= if cons, else the
        string itself)
     3. Calls =elfeed-feed-autotags(url)= (Elfeed built-in)
     4. Checks if =elfeed-translate-feed-tag= is =memq= in the
        autotags list

     =elfeed-feed-autotags= converts keyword symbols (=:translate-title=)
     to plain symbols (=translate-title=) via
 =elfeed-keyword->symbol=, so both formats work.

  ** elfeed-org Compatibility

     In =elfeed-org=, org headlines use colon-tag syntax
     (=:translate_title:=).  When =elfeed-org= parses the org file,
     these become autotags on the feed.  =elfeed-feed-autotags=
     converts them to plain symbols.  Our =memq= check then matches.

     The default =elfeed-translate-feed-tag= is =translate_title=
     (underscore) to match org-mode tag conventions, but users can set
     it to =translate-title= for direct =elfeed-feeds= usage.

* show-feeds Buffer

  ** Dual Output Modes

     | Condition            | Format         | Mode          |
     |----------------------+----------------+---------------|
     | =elfeed-org= loaded  | Org headlines  | =org-mode=    |
     | =elfeed-org= absent  | Elisp forms    | =emacs-lisp-mode= |

  ** Org Format

     - Level-2 heading (=**=) for "Translated feeds" group
     - Level-3 headings (=***=) for individual feeds
     - Uses org link syntax: =[[file:///...][[TL] Title]]=
     - Properties drawer with =:ORIGINAL-FEED:= for traceability
     - Drawers folded on initial display (=org-fold-hide-drawers-all=)

     This format sits under the user's existing level-1 feed group in
     their elfeed-org file.

  ** Buffer Lifecycle

     - Buffer name: =*elfeed-translate-feeds*=
     - Always regenerated fresh (content is erased and rebuilt)
     - RSS files are regenerated before displaying
     - Read-only mode enabled after population

* Known Bugs Fixed (Historical Record)

  1. *JSON parse: hash-table vs alist* — =json-parse-string= ignores
     dynamic variable =json-object-type= when ANY keyword argument is
     passed.  Fix: always pass =:object-type 'alist= explicitly.

  2. *batch chain deadlock* — =elfeed-translate--busy= was cleared in
 =unwind-protect= cleanup /after/ the callback.  The callback
     triggers the next batch, which checks =busy= and skips.  Fix:
     clear =busy= /before/ invoking the callback.

  3. *=cl-return-from= failing* — =cl-return-from= inside
 =condition-case= error handler couldn't find the =cl-block=
     established by =defun=.  Fix: replaced with explicit
 =catch='parse-error / =throw='parse-error=.

  4. *Prompt template= *— 3 =%s= placeholders but =format= only got 1
     argument.  Fix: reduced to 1 =%s= and used natural language for
     remaining references.

* Future Improvements (Ideas)

  - *Cache key by entry GUID* instead of title string — more robust
    against minor title changes (e.g., RSS feed adding/removing a
    trailing space)
  - *auto-refresh after translation* — call =elfeed-update-feed= on
    local files with a timer without triggering recursive hooks
  - *per-feed target language* — some feeds → English, others →
    Chinese, controlled by a feed-level property
  - *translation progress indicator* — show a progress bar or spinner
    during batch processing
  - *error retry* — retry failed batches with exponential backoff
  - *streaming API support* — use SSE to get incremental results

* Testing

  Since this package interacts with external APIs and Elfeed's
  database, testing is best done interactively:

  1. =M-x eval-buffer= on =elfeed-translate.el=
  2. =(setq elfeed-translate-debug t)=
  3. =M-x elfeed= → verify "Setup — N feed(s), M cached" message
  4. =M-x elfeed-translate-stats= → verify feeds and cache counts
  5. =M-x elfeed-update= → observe batch processing in =*Messages*=
  6. Verify =~/.elfeed/translated/*.xml= files are generated with
     correct content
  7. Run =elfeed-update= again → verify "All titles up to date"
     (cache is working)
